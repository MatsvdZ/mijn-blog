Kilian Valkhof

Datum: 04-02

Onderwerp: Stop using JS for that

Moving features from JS to CSS & HTML

Tijdens de Weekly Nerd sessie van Kilian Valkhof stond er vooral één centraal idee centraal: probeer zo min mogelijk JavaScript te gebruiken wanneer HTML en CSS hetzelfde probleem kunnen oplossen. Kilian liet zien hoe veel features die vroeger alleen met JavaScript mogelijk waren, nu ook met HTML en CSS gedaan kunnen worden.

The Rule of Least Power

The Rule of Least Power is een principe dat stelt dat je altijd de minst krachtige taal moet gebruiken die nog steeds in staat is om het probleem op te lossen. Volgens Kilian zijn HTML en CSS vaak beter geschikt dan JavaScript, omdat ze declaratief zijn. Je beschrijft wat je wilt bereiken, in plaats van hoe je het moet doen. Hierdoor wordt code eenvoudiger, sneller en beter onderhoudbaar.

Browsermakers are listening

Ook vertelde Kilian dat browsermakers steeds meer luisteren naar wat developers bouwen. Nu worden features die developers zelf maken steeds vaker native ondersteund door browsers. Een voordeel dat hierbij past is dat de technologie vrijwel altijd backwards compatible blijft. De code die dus vandaag werkt, zou in de toekomst ook nog steeds moeten werken. Hierdoor had ik wel een bijzondere gedachte, namelijk: Alleen omdat ik weet hoe iets met JavaScript kan, betekent dat niet automatisch dat ik het ook met JavaScript moet doen., omdat dit misschien niet de beste oplossing is.

Moderne CSS & HTML features

Kilian liet ons ook een paar voorbeelden zien van moderne CSS en HTML features. Zo heeft hij ons laten zien hoe je tegenwoordig veel makkelijker custom switches kunt maken en het stijlen van checkboxes en radio buttons veel makkelijker is geworden. Normaal bepaalt de browser dit, maar met appearance: none; kun je die styling verwijderen, en zelf ontwerpen. Ook accessibility speelde hierbij een grote rol, want met :focus-visible kun je toetsenbordgebruikers een duidelijke focus geven, zonder dat dit de ervaring van muisgebruikers verpest.

Verder heeft hij het nog gehad over smooth scrolling en in-page transitions. Ook details, summary en dialog werden nog besproken.

Deze talk heeft mij iets anders laten nadenken over frontend development. Ik kijk dus niet meteen naar JavaScript, maar ik kijk eerst of HTML en CSS het probleem al kunnen oplossen. Het was interessant om te zien hoeveel moderen features er tegenwoordig zijn, en dat dit makkelijker, sneller en §beter is dan JavaScript.

Peter-Paul Koch

Datum: 19-02

Onderwerp: Browser and support detection: Geschiedenis van browsers

Tijdens deze Weekly Nerd sessie vertelde Peter-Paul Koch over browsers, rendering engines en hoe browsers omgaan met support en compatibility. Ik vond het vooral interessant om meer te leren over wat er eigenlijk gebeurt voordat een website zichtbaar wirdt op je beeld.

Verschillende onderdelen

Koch legde uit dat een browser uit verschillende onderdelen bestaat, zoals HTML- en CSS-parser, de DOM, de AOM en een JavaScript engine. Vooral het verschil tussen JavaScript zelf en de DOM vond ik interessant. JavaScript is een programmeertaal, terwijl de DOM zorgt dat JavaScript elementen op een pagina kan aanpassen.

Rendering & performance

Koch gaf ook een beetje uitleg over rendering en performance. Zo vertelde hij dat layoutberekeningen zwaar zijn voor browsers, daarom is het belangrijk om layouts simpel te houden. Zijn advies was als volgt: "Keep it stupid simple". Dat neem ik mee, omdat ik zelf soms geneigd ben im dingen onnodig complex te maken.

Render blocking

CSS en JavaScript kunnen het laden van een pagina vertragen. vooral het verschill tussen defer en async vond ik handig om te weten.

  • Defer voert scripts uit nadat de HTML geladen is.
  • Async laat de browser zelf bepalen wanneer het script uitgevoerd wordt.

Backward compatibility

Browsers moeten vrijwel alles wat ooit heeft gewerkt blijven ondersteunen. Daardoor werken dus ook oudere HTML features nog. Dit vond ik interessant, omdat browsers daardoor dus met veel oude technieken moeten blijven werken.

Browser geschiedenis

Op het einde vertelde Koch nog wat over de geschiedenis van browsers zelf, wat ik ook erg interessant vond. Hij vertelde over de eerste browser van Tim Berners-Lee tot de moderne browsers zoals Chrome en zelfs wat nieuwere broswers zoals Ladybird. Wat ik uit deze sessie heb gehaald, is meer begrip voor hoe browsers werken en hoe belangrijk performance en makkelijke code is.

Nils Binder

Datum: 04-03

Onderwerp: CSS to the rescue

Tijdens deze Weekly Nerd sessie vertelde Nils Binder over design, CSS en moderne manieren van webdevelopment.

Waar heeft hij aan gewerkt?

Nils werkt bij 9elements, waar ze werken met verschillende units, namelijk:

  • Communication Design
  • Product Development
  • Web Development

Hij vertelde over andere projecten waar hij aan heeft gewerkt, zoals de app van FC Schalke 04 (erg leuk om te horen als voetbal fan) en een accessibility project voor OECD. Hier ging het vooral om kleurgebruik en toegankelijkheid, zodat informatie en grafieken duidelijk blijven voor iedereen.

Then and now in webdesign

Een onderwerp wat ik erg interessant vond, was het verschil tussen vroeger en nu binnen webdesign. Vroeger ontwierpen designers vaak in Adobe Photoshop, terwijl nu bijna iedereen Figma gebruikt. Volgens Nils lijkt Figma veel meer op hoe CSS daadwerkelijk werkt, met dingen zoals flecbox, gap, padding en variabelen. Daardoor sluiten design en development beter op elkaar aan.

Layouts

Ook liet Nils zien hoe belangrijk verhoudingen en layouts zijn. In plaats van alleen naar pixels te kijken, moet je kijken naar de verhouding tussen elementen. Hij gebruikte voorbeelden met CSS grid en fr units om layouts flexibel en responsive te maken. Dit vond ik interessant, omdat ik zelf vaak nog moeite heb met het maken van goede layouts.

Ook animaties kwamen nog voorbij, zoals het gebruik van steps(). Hij vertelde ook nog even over subgrid en view transitions. Vooral de view transitions vond ik interessant, omdat dit een nieuwe manier is om soepele animaties te maken tussen pagina's zonder JavaScript.

Wat ik dus uit deze sessie heb gehaald, is dat CSS steeds sterker wordt en dat design en development steeds dichter naar elkaar toe groeien. Ook ben ik dus meer gaan nadeken over layouts en verhoudingen in plaats van alleen maar naar pixels te kijken.

Robert Boersma & Yolijn van der Kolk

Datum: 12-03

Onderwerp: FF formulieren maken met NL design System richtlijnen

Tijdens deze Weekly Nerd sessie vertelden Robert Boersma en Yolijn van der Kolk over het ontwerpen van formulieren volgen de richtlijnen van het NL Design System. De sessie draaide vooral om gebruiksvriendelijkheid en toegankelijkheid. Wat mij meteen opviel, was hoeveel details belangrijk zijn bij het maken van een goed formulier.

De opbouw

Er werd eerst uitgelegd hoe een formulier logisch opgebouwd moet zijn. Dit moet met een duidelijke introductie, de noodzakelijke informatie, gebruikers hun gegevens laten controleren en na het verzenden een duidelijke bevestiging geven. Ook werd er benadrukt dat het fijn is als gebruikers tussentijds een formulier op kunnen slaan.

Daarnaast ging het veel over toegankelijkheid. Labels moeten altijd zichtbaar zijn en gekoppeld worden met een label. alleen placeholder tekst is dus niet voldoende. Ook foutmeldingen en beschrijvingen moeten gekoppeld zijn aan invoervelden met bijvoorbeeld aria-describedby, zodat een screenreader deze goed kan voorlezen.

Verder kwamen er vooral veel praktische designregels voorbij, zoals: Componenten moeten voldoende kleurcontrast hebben, knoppen moeten groot genoeg zijn (bv. minimaal 44x44 pixels). Ook de flow van het formulier moet logisch zijn, zoals consistente navigatie en zichtbare voortgang (stap 2 van 4).

Wat ik vooral uit deze sessie heb gehaald is dat een formulier meer is dan alleen een paar invoervelden. Kleine keuzes die je maakt voor labels, foutmeldingen, navigatie en screenreaders maken een groot verschil voor gebruikerservaring en toegankelijkheid.

Rosa

Datum: 02-04

Onderwerp: Hackers United

Tijdens deze Weekly Nerd verteld Rosa over hacking, onafhankelijkheid van grote tech-bedrijven en het belang van zelf controle houden. De sessie was wat korter en ging minder over traditionele frontend development en juist wat meer over alles daarachter.

Wat is een hacker

Ze opende haar talk met de vraag: wat is een hacker eigenlijk? Het ging niet alleen over iemand die systemen kraakt, maar vooral over mensen die begrijpen hoe systemen werken en zelf oplossingen proberen te bouwen. Een belangrijk punt was dat je zoveel mogelijk zelf moet proberen te maken, zodat je hier zelf de controle over houdt.

Planned obsolescence

Door Planned obsolescence worden apparaten sneller vervangen in plaats van gerepareerd. Daardoor groeit de afhankelijkheid van grote techbedrijven. Dit geldt niet alleen voor consumenten, maar ook voor bedrijven en overheden.

Wat ik interessant vond, was hoe ze keek naar oplossingen die low tech waren. Zo liet ze een voorbeeld zien van projecten die draaien op oude hardware, zoals een computer uit 2007 die uit het afval was gehaald. Hierdoor moet je dus anders nadenken over hoe je bijvoorbeeld websites bouwt. Zware frameworks werken minder goed, waardoor simpele en efficiënte oplossingen belangrijker worden.

Static site generators

Ook kwamen static site generators voorbij, zoals Eleventy (11ty). Volgens Rosa hebben die veel voordelen: minder belasting op servers, leesbare code en als alles faalt heb je altijd nog de markdownbestanden. Dat idee vond ik interessant, omdat het laat zien hoe belangrijk simpele en duurzamen technologie eigenlijk kan zijn.

Wat ik vooral uit deze sessie heb gehaald, is dat technologie niet persé nieuwer of complexer hoeft te worden. Soms zijn simpele en lichte oplossingen sterker en duurzamer. Ook ben ik meer gaan nadenken over hoeveel controle je uit handen geeft wanneer je afhankelijk bent van grote platforms en frameworks.

Johann Huijkman

Datum: 09-04

Onderwerp: Accessibility

Tijdens deze Weekly Nerd vertelde Johann Huijkman over toegankelijkheid en waarom toegankelijk ontwerpen verder gaat dan alleen de WCAG-richtlijnen. Hij werkte jarenlang bij Q42 en is later overgestapt naar Stichting Accessibility.

Theorie VS praktijk

Een belangrijk punt uit de sessie was de iets theoretisch toegankelijk kan zijn, maar in de praktijk alsnog lastig te gebruiken blijkt. Daarom is het testen met echte gebruikers volgens hem veel belangrijker dan een checklist volgen. Ook vertelde hij dat de WCAG-richtlijnen vaak verkeerd geïnterpreteerd worden. Bijvoorbeeld: niet alles hoeft een alt-tekst te hebben. Decoratieve afbeeldingen hoeven dat juist niet.

Screenreaders

Ook heeft hij het gehad over screenreaders. Die hebben al heel veel ingebouwde functionaliteiten en shortcuts, waardoor gebruikers niet altijd de hele pagina hoeven door te lopen. Hierdoor besefte ik me dat je goed moet testen voordat je zelf extra oplossingen probeert te bedenken.

De gebruikers

Tijdens deze sessie liet Johann drie verschillende gebruikers zien als voorbeeld. Elke gebruiker had een net wat andere beperking. Het interessante hieraan was om te zien dat er best veel verschil zit tussen de soort toegankelijkheid die een persoon nodig kan hebben. Bij bijvoorbeeld Martijn (slechtziend) ging het vooral om schaalbaarheid, contrast en duidelijke focusindicatoren. Bij Naduah (mobiliteitsproblemen & prikkelgevoelig) zijn vooral rustige layouts belangrijk, en weinig handelingen. Ook het opslaan van gegevens is belangrijk, omdat lange processen vermoeiend zijn. En Guido (autisme) raakt snel overprikkeld door teveel informatie en visuele effecten. Hier zijn duidelijke instructies, voorspelbaarheid en een rustige interface belangrijk.

Wat ik dus vooral uit deze sessie heb gehaald, is dat toegankelijkheid niet één vaste oplossing heeft. Verschillende gebruikers hebben compleet andere behoeften. Hierdoor moet je als developer geen aannames maken over wie jouw website gebruikt en hoe die gebruikt wordt.

LEVEL 02

SPRINT 0

Mijn startpunt, aanpak en eerste inzichten uit de beginfase.

Link naar live site: Dit ben ik

Sprint 0

Tijdens sprint 0 heb ik een persoonlijke website gemaakt die als een soort introductie van mijzelf en mijn interesses dient. In het begin had ik een heel ander concept met een landschap en een weg waar ik als het ware overheen reed met mijn motor. Maar tijdens het ontwerlen heb ik besloten om dit helemaal om te gooien naar een futuristische Tron-achtige omgeving met felle neonkleuren en animaties. Ik heb ook voor het eerst gewerkt met horizontaal scrollen zodat de verschillende delem van de site beter samen kwamen. Daarnaast heb ik geëxperimenteerd met animaties zoals glitcheffecten, hover animaties, pulsing neon tekst animaties en een custom cursor. Ook heb ik geoefend met data uit de FDND API dynamisch inladen. Hierbij heb ik er ook voor gezorgd dat je andere mensen in de database kan opzoeken als zogenaamde 'players.' Verder heb ik nog een light en een darkmode gemaakt, muziek toegevoegd en veel aandacht besteed aan responsiveness en het netjes structureren van mjin code.

Mijn nieuwe inzichten

Wat ik vooral uit sprint 0 heb gehaald, is hoe belangrijk een sterk concept en een sterke basis zijn, voordat je begint met coderen. Ik merkte dat mijn project pas goed begon te werken toen ik een duidelijke stijl had gekozen en een idee had van de layout die ik wou gebruiken. Daarnaast heb ik redelijk wat geleerd over animeren met CSS, responsiveness en het combineren van vormgeving en interactie. Ook besefte ik me dat experimenteren een belangrijk onderdeel is van het process. Door te experimenteren kwam ik uit bij mijn uiteindelijke concept. Veel van wat ik aan het begin wilde lukte eerst niet, of kwamen niet helemaal lekker samen. Maar door steeds verschillende dingen te proberen kwam ik steeds verder. Daarnaast ben ik iets bewuster gaan nadenken over de kwaliteit, structuur en consistentie van de code die ik schrijf.
